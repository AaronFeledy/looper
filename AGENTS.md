# AGENTS.md

This file is for non-obvious repo context only. Keep it short and current.

## Stack

- Bun + TypeScript, no bundler. `tsc -b` typecheck only, `noEmit: true`.
- Runtime deps: `@opentui/core`, `@opencode-ai/sdk`, `yaml`. No other production deps; do not add them lightly.

## Commands

- `bun install`
- `bun run typecheck`
- `bun test` &mdash; includes a real-network e2e against opencode + gpt-5.5; self-skips when `opencode` is not on `PATH`. ~50 seconds, ~zero cost.

## Architecture

- `src/main.ts` &mdash; CLI entry. Decides TTY vs non-TTY mode, owns the renderer, spawns/attaches the opencode server, and wires `src/engine/run-engine.ts`.
- `src/engine/run-iteration.ts` &mdash; runs one iteration: loop over steps, handle restart / skip / failure retries / background-task waits. `src/lib/orchestrator.ts` is a compat re-export shim. `runIteration` accepts an optional `resume` (`{sessionID?, messageID?, stepName?, promptText?, looperMessageIDs?}`) applied to the first processed step: if the recorded session is still `pending` and a `messageID` exists it reattaches via `reattachOpenCodeStep`; `messageID` selects the user turn whose assistant result determines the outcome, while `looperMessageIDs` independently identifies Looper-authored user turns hidden from output. A verified `idle` session normally restarts the step fresh, but an explicit manual recovery-nudge reuses that idle failed session while preserving the prompt and Looper-owned message IDs. For any other state (`unknown`, `pending`-without-`messageID`, or a step-name mismatch) it fails CLOSED &mdash; `stopServerSession` must confirm the old session is stopped before a fresh one is created, never proceeding on an unconfirmed stop (matches the no-overlapping-generations guard).
- `src/lib/prompt-context.ts` &mdash; pure builder for the `<looper-context>` block prepended to every step prompt. `src/engine/run-iteration.ts` composes it fresh immediately before each `runOpenCodeStep` call (the single prompt-send call site, so fresh/nudge/restart/retry/background-continuation prompts all get it for free) using a dedicated `completedLogicalSteps` ledger keyed by `stepIndex` &mdash; never `state.steps` rows and never the pre-existing `completed` array &mdash; so duplicate step names stay distinct entries and the current step's own retry/restart attempts never appear as a "prior step" of itself. Agents never see past-iteration data (the "iteration N of M" counter is the only trace of the past). The ledger is seeded on resume from `RunState.stepSessions` (see the `run-state-store.ts` note below), filtered to `stepIndex < startStepIndex` so an in-flight step's own orphaned session can't leak into its own first prompt after a crash/resume. `context:` policy (see `config.ts`'s `resolveContextPolicy`) is resolved per step and can disable any of the seven sections.
- `src/opencode/*` &mdash; runs steps against opencode (`step-runner.ts`), reattaches/evaluates prior sessions (`reattach.ts`), supervises SSE event streams with watchdog/resubscribe/backfill (`event-stream.ts`), manages background continuation records (`background-tasks.ts`/`continuation-records.ts`), session health (`session-health.ts`), and opencode IDs (`opencode-id.ts`). `src/lib/runner.ts` is a compat re-export shim. Every prompt is sent with a self-generated opencode `msg_ŌĆ”` id (see `createOpencodeID`) so failures can be correlated against the actual assistant message via `session.messages()`. The event subscription is supervised: a watchdog re-subscribes (and backfills missed output via `session.messages()`) when the SSE stream ends or stalls while `session.status` still reports the session busy &mdash; the prompt POST and the event stream are on separate `AbortController`s so reconnecting events never disturbs the in-flight prompt.
- `src/lib/state.ts` &mdash; in-memory TUI state, listener pub/sub.
- `src/persistence/run-state-store.ts` &mdash; run-state write facade over `src/lib/state-files.ts` stop/resume files under the config dir. **Must call `initStatePaths({ configDir })` before state-files exports are used.** `.looper-resume-step.json` is the legacy step-only checkpoint (kept for back-compat); `.looper-run.json` (`RunState`: `{iteration, stepIndex, stepName, sessionID?, messageID?, promptText?, looperMessageIDs?, title?, looperRunID?, stepSessions?}`) is the authoritative iteration-aware resume pointer. `looperRunID` is generated per logical Looper run, persisted across resume, and sent in opencode session metadata for Looper-owned sessions. `sessionID`/`messageID`/`promptText`/`looperMessageIDs` are written only while a step is in flight (via `runOpenCodeStep`'s `onSessionBound`, fired right before `session.prompt`, or when OpenCode posts a continuation turn); `messageID` is the current outcome-classification turn, `promptText` is the exact Looper prompt shown by the prompt dialog, and `looperMessageIDs` is the ownership set used to hide only Looper-authored user turns. On step `done` the pointer is advanced to the next step (or next iteration) WITHOUT those in-flight fields, so `--continue` never reattaches to an already-completed step. Old files without prompt metadata remain valid and treat their in-flight `messageID` as the sole known Looper-owned ID. `title` is the iteration's generated work-description: it rides along with the pointer across step advances (so a resumed run rehydrates it into `runIteration`'s `initialWorkDescription` and steps that only inherit the title still apply it to their fresh opencode sessions instead of letting opencode auto-title from the prompt) and is dropped when the pointer crosses into a new iteration. `stepSessions` (`{stepIndex, stepName, sessionID}[]`) follows the exact same lifecycle as `title` (upserted at session bind, carried across step advances within the iteration, dropped at the iteration boundary) and feeds the prompt-context session list on resume &mdash; see the `prompt-context.ts` note below. Resuming is the default; both files are cleared together on `--fresh` (or when a run reaches `max_iterations`). A resume pointer whose iteration exceeds `max_iterations` falls back to a fresh run.
- `src/lib/session-metadata.ts` &mdash; builds structured metadata for Looper-owned opencode sessions (`looper`, `looperRunID`, iteration/step, config/repo dir, purpose, optional parent session). Step sessions get `purpose: "step"`; throwaway title sessions get `purpose: "title"` plus `parentID`/`parentSessionID`.
- `src/lib/history-stream.ts` &mdash; current-run, in-RAM iteration history. `snapshotIterationToHistory` (called at the top of `resetIterationState` in `main.ts`, before it wipes `state.steps`) pushes the outgoing iteration's step rows (name/status/sessionID/title/prompt ownership metadata) into `state.history`. When `state.historyView` is set and the selected historical step has a `sessionID`, the streamer refetches `session.messages(sessionID)` once per selection and passes that step's `looperMessageIDs` to `renderSession`, so plugin/server user turns remain visible while Looper prompts stay hidden. The prompt dialog resolves the selected historical snapshot's exact `promptText`. History is RAM-only (lost on exit); resume across exits is the `.looper-run.json` path, not history.
- `src/lib/fallback.ts` &mdash; non-TTY linear runner.
- `src/lib/config.ts` &mdash; `looper.yaml` loader; resolves `prompt:` paths relative to config dir.
- `src/config/tunables.ts` &mdash; owns `LOOPER_*` timeout/default parsing.
- `src/lib/sdk-server.ts` &mdash; spawns `opencode serve --port=0`, captures the listening URL from stdout. Has its own timeout; do not block tests on it.
- `src/lib/title.ts` &mdash; throwaway-session title generation. Defaults the title session's `agent` to `looper-title` (see `title-agent.ts`); the title prompt is still passed as a `system` override and the model is still chosen per-provider by the cheap-model heuristic. Naming the agent is what stops opencode applying the default agent's adaptive-thinking variant to reasoning-capable cheap models (which reject it with a 400). The cheap-model heuristic (`resolveHeuristicTitleModel`) prefers non-reasoning models but, for a reasoning-only provider (e.g. openai gpt-5.x), falls back to the cheapest reasoning model rather than returning `undefined` &mdash; returning `undefined` lets opencode pick its heavyweight global default (this is what once ran the title agent on opus). Do not add provider-specific model workarounds here for request-shape bugs; if a plugin injects invalid Anthropic thinking fields, fix that plugin so opencode/model metadata remains the source of truth. The whole create+prompt is bounded by `LOOPER_TITLE_GEN_TIMEOUT_MS` (default 60s); on timeout the title prompt is aborted so a misbehaving title agent can't run unbounded. The throwaway title session is ALWAYS aborted+deleted in the `finally` (success or failure) &mdash; an earlier "keep failed title session for review" path leaked sessions that each held a copy of the step work-log, and the failure is already logged, so it was rolled back.
- `src/lib/attached-server-agents.ts` &mdash; attach-mode preflight checks. In addition to managed agent validation, `assertAttachedServerLocation` best-effort checks the attached server's active location (prefer `client.v2.location.get`, fallback `client.path.get`) and fails early on repo-dir mismatch; if the SDK/server cannot report a location, attach behavior falls back to the managed-agent check.
- `src/lib/title-agent.ts` / `src/lib/opencode-managed-resources.ts` &mdash; materializes the hidden `looper-title` opencode subagent (`mode: subagent`, `hidden: true`, no `variant`) into opencode's GLOBAL agent dir. `main.ts` calls `ensureTitleAgent()` before the server starts. The agent frontmatter disables ALL tools (`tools: { "*": false }`) and denies tool permissions (`permission: { "*": deny, ... }`): it is fed a step's assistant work-log as its user message, and with tool access a capable model reads that as a task list and starts executing it (an observed opus runaway ran ~24 turns over 8 minutes). The disabled tools force a single text turn; `LOOPER_TITLE_GEN_TIMEOUT_MS` is the backstop for the `--attach` path where a running server may not have reloaded the locked-down agent file.
- `src/lib/event-consumer.ts` / `src/core/events.ts` / `src/presentation/legacy-line-format.ts` &mdash; event pipeline: SDK message/part events become the structured `LooperEvent` union, then golden-tested legacy text lines for terminal output.
- `src/engine/run-engine.ts` &mdash; shared loop for TTY and non-TTY frontends: resume plan, run-state advancement, stop handling, iteration invocation, and recovery retry selection.
- `src/engine/adjudication-routing.ts` / `src/persistence/adjudication-store.ts` / `src/lib/adjudication-files.ts` &mdash; PRD oscillation adjudication. State lives in `configDir` as `.looper-adjudicate` (marker + reason text), `.looper-prd-history.json` (per-story `passes` transition log plus an `adjudicatedThrough` watermark) and `.looper-adjudicate-session.json` (the in-flight adjudicator session), owned by these files, NOT `state-files.ts` directly. `readPrdPasses` is per-story tolerant and semantically aligned with the canonical reader (`src/lib/prd.ts`): `passes === true` means passing, unusable stories (no id) are skipped rather than discarding the whole snapshot, so one malformed story can't silently disable detection. Detection runs only over the ACTIVE window (`readActiveHistory` = records after `adjudicatedThrough`): a story's `passes` flips trueŌåÆfalse at least `prdFlipThreshold` times (`LOOPER_PRD_FLIP_THRESHOLD` env, else `prdFlipThreshold:` in `looper.yaml`, else default 2; resolved in `src/config/tunables.ts`). A completed adjudication calls `markAdjudicated` to advance the watermark, so resolved flips (and the adjudicator's own transitions) never retrigger while the full trail is retained for forensics. The optional top-level `adjudicate:` block in `looper.yaml` (same schema as one `steps:` entry, default name `adjudicate`, loaded by `loadAdjudicateStep` in `config.ts`; validated at startup by `ensureConfigValid` in `main.ts`) is never part of `loadSteps()`. Routing (`decideRouting` in `run-iteration.ts`, checked both mid-iteration and at iteration start so a stale/resumed marker routes before step 0): a detected or agent-written marker skips the remaining configured steps and runs the adjudicate step as one ephemeral dynamic row (`insertAdjudicationRow`) whose prompt is prefixed with the marker reason via `withAdjudicationReason`. The adjudicator's session is persisted (via `writeSession` in `onSessionBound`, before prompt dispatch) so a crash mid-adjudication is reconciled (confirm-stopped) at the next iteration-start routing before a fresh adjudicator launches &mdash; no overlapping generations. Only a `done` adjudicator clears the marker/session and advances the watermark; a non-`done` adjudicator FAILS CLOSED (marker retained for re-route, session confirm-stopped, `StepFailureError` thrown), never treated as resolved. No `adjudicate:` step configured: writes the stop file with the oscillation reason instead. The adjudicate row is never a resumable position (no run-state pointer entry); the marker + session record are the durable resume signals. `readPrdHistory` never overwrites a present-but-corrupt history: `appendPrdHistory` quarantines it to `.looper-prd-history.json.corrupt-<ts>` (or skips the write if it can't) rather than truncating forensic data. History/session clear only on `--fresh`, never via `clearRunArtifacts` (survive normal run-end cleanup).
- `src/watchers/*` &mdash; branch/GitHub/PRD watchers and branch-delta helpers.
- `src/tui/*` &mdash; opentui renderables. TUI output blocks are built from structured events via `src/presentation/tui/stream-blocks.ts`; the contract is the `LooperEvent` union plus the golden-tested legacy formatter, not a string parser.

## Gotchas

- Path semantics: `repoDir = process.cwd()`. `configDir` is resolved (in `main.ts`) from `--config-dir`, else `LOOPER_CONFIG_DIR`, else the first of `$PWD/.looper`, `$PWD/.local/looper`, `$PWD/.local/.looper` that already holds a config file, else defaults to `$PWD/.looper`. State files live in `configDir`. The CLI auto-creates `configDir` but fails with exit 2 if no config file is present. Config file resolution prefers `looper.yml`, then `looper.yaml`, `.looper.yml`, `.looper.yaml` (see `CONFIG_FILE_NAMES`/`findConfigFile` in `config.ts`).
- `bin/looper` is a bash wrapper that resolves symlinks itself; symlink it from `~/.local/bin/looper` and it still finds `src/main.ts`.
- `noUncheckedIndexedAccess: true` is on. `frames[i]` from a literal array still needs `!`.
- `src/opencode/background-tasks.ts`'s `LOOPER_CONTINUATION_EXIT_GRACE_MS` window decides how long to wait after an opencode step ends for a background-task record. Don't shorten it in tests unless you mock the file.
- `event-consumer.ts` debug logs gate on `LOOPER_DEBUG_EVENTS=1`.
- `ensureTitleAgent()` (`title-agent.ts`) writes `looper-title.md` into opencode's GLOBAL agent dir (`$XDG_CONFIG_HOME/opencode/agent`, else `~/.config/opencode/agent`) at startup &mdash; a side effect outside `configDir`/`repoDir`. It only overwrites a file carrying its own managed marker, never a user-authored `looper-title.md`. Caveat: it's written before looper *spawns* opencode, so the spawn path picks it up at boot; on the `--attach`/`attachUrl` path the already-running server may not have loaded it, in which case title gen degrades to no-title (best-effort) rather than breaking the loop.
- `src/opencode/event-stream.ts`'s event watchdog is tunable via `LOOPER_EVENT_WATCHDOG_POLL_MS` (default 15s), `LOOPER_EVENT_STALL_MS` (default 45s; how long no events may pass before it probes `session.status`), and `LOOPER_EVENT_RESUBSCRIBE_BACKOFF_MS` (default 1s floor between reconnects). A stream that ends cleanly triggers an immediate probe regardless of the stall window. `createSessionEventConsumer` keeps print/dedup state across reconnects so resubscribe + `backfill()` never double-prints.
- The e2e test uses `agent: build` + `model: openai/gpt-5.5` + `variant: low`. Reasoning-only models (e.g. claude-haiku-4-5) reject the build agent's adaptive-thinking variant with a 400 &mdash; if you change the model, also align the agent/variant.
- Test scratch dirs land under `test/.tmp/` and are auto-cleaned unless `LOOPER_E2E_KEEP=1`.
